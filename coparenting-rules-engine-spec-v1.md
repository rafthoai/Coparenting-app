# Co-Parenting App Rules Engine Specification v1

## Purpose

This document defines the deterministic rules engine that powers custody schedule generation, holiday overrides, school-day handling, swap application, approvals, and schedule conflict resolution in the co-parenting app.

The rules engine exists to transform confirmed user inputs and reviewed AI-assisted extractions into a reliable calendar and responsibility model. AI may help extract or propose rules, but the rules engine is the system of record that applies them consistently.

## Core Principle

AI should **read and propose**.
The rules engine should **enforce and apply**.

The engine must be:
- deterministic
- explainable
- auditable
- override-aware
- safe in ambiguous scenarios

---

# 1. Scope of the Rules Engine

The rules engine should handle:
- recurring custody schedules
- alternating patterns
- holiday overrides
- school break overrides
- teacher planning days and no-school day assignment
- handoff timing generation
- approved swap application
- precedence rules across overlapping schedule logic
- child-specific responsibility assignment
- schedule regeneration when rules change

The rules engine should **not** initially handle:
- legal interpretation on its own
- fairness scoring
- payments or reimbursement logic
- emotional mediation
- advanced predictive behavior
- unresolved ambiguous clauses without user confirmation

---

# 2. Key Concepts

## 2.1 Schedule Layers

The child’s final schedule should be generated by applying rules in layers.

### Layer A: Base Custody Pattern
The default recurring custody arrangement.
Examples:
- every Monday/Tuesday with Parent A
- alternating weekends with Parent B
- 2-2-3 rotation
- week-on/week-off

### Layer B: Holiday Overrides
Special date ranges that replace the base custody pattern.
Examples:
- Thanksgiving alternates yearly
- Christmas split into segments
- Mother’s Day always with mother

### Layer C: School and Break Overrides
Rules triggered by school-specific dates.
Examples:
- spring break with Parent A in odd years
- teacher planning days follow the parent whose day it would otherwise be
- no-school Mondays always belong to Parent B

### Layer D: Approved Swaps and One-Off Exceptions
User-approved schedule changes that override generated schedule entries.
Examples:
- one-time weekend swap
- partial-day exchange adjustment
- approved holiday exception

### Layer E: Manual Admin Overrides
A user-confirmed, explicit override inserted into the schedule when needed.
This should be rare and logged clearly.

### Precedence Order
The engine should apply higher-precedence layers last.

Recommended precedence:
1. Base custody pattern
2. Holiday overrides
3. School and break overrides
4. Approved swaps and one-off exceptions
5. Manual admin overrides

# 4. Source Data Inputs

The rules engine should only consume confirmed inputs from trusted sources.

## Allowed Inputs
- manually configured custody setup
- confirmed agreement extraction output
- confirmed school calendar import output
- approved swap requests
- approved one-off exceptions
- confirmed handoff settings

## Disallowed Direct Inputs
- unreviewed AI extraction output
- raw user messages
- ambiguous legal clauses without confirmation
- unapproved suggested edits

# 6. Base Custody Logic

## 6.1 Weekly Fixed-Day Assignment

Example:
- Parent A has Monday and Tuesday
- Parent B has Wednesday and Thursday
- weekends alternate

The engine should:
1. expand weekly recurrences into schedule segments
2. merge contiguous segments owned by the same caregiver when appropriate
3. insert handoff events when caregiver changes

## 6.2 Alternating Weekend Logic

For alternating weekends, the engine must define:
- anchor start date
- what counts as the first weekend
- weekend time boundaries (e.g., Friday 5 PM to Sunday 6 PM)
- relationship to weekday baseline rules

Without an explicit anchor date, the engine should not guess. It should require confirmation.

## 6.3 Rotating Patterns

Support common patterns such as:
- 2-2-3
- 2-2-5-5
- week-on/week-off

Represent these as pattern arrays with an anchor date.

Example conceptual structure:
- pattern_units: [2,2,3]
- caregiver_order: [A,B,A]
- anchor_date: 2026-01-05

The engine should cycle through pattern units deterministically.

---

# 7. Holiday Override Logic

## 7.1 Holiday Rule Behavior

Holiday rules replace the base custody pattern for their defined interval.

Examples:
- Thanksgiving Day 9:00 AM to Friday 9:00 AM
- Christmas split into Part A and Part B
- Mother’s Day 8:00 AM to 8:00 PM

## 7.2 Holiday Assignment Strategies

Supported strategies should include:
- fixed caregiver
- alternating by year parity
- alternating by explicit sequence
- split interval between caregivers
- inherit from base schedule if no explicit override exists

## 7.3 Collision Handling

If multiple holiday overrides apply to the same interval:
1. use explicit priority values
2. if still tied, prefer narrower/specific rule over broad one
3. if still tied, flag conflict for manual resolution

## 7.4 Holiday Definitions

The system should maintain a configurable holiday catalog by locale where useful, but users must be able to confirm or customize actual observed date windows.

---

# 8. School and Break Logic

## 8.1 School Calendar as Event Layer

The school calendar should be imported as date-typed events, such as:
- no_school_day
- teacher_planning_day
- early_release_day
- winter_break
- spring_break
- school_closure

The rules engine should then apply SchoolBreakRule logic against those dates.

## 8.2 Assignment Strategies for School-Day Exceptions

Supported strategies should include:
- fixed caregiver
- follow base schedule owner
- alternate by year parity
- alternate by event occurrence count
- attach to adjacent weekend owner
- manual confirmation required

## 8.3 Break Interval Handling

For multi-day breaks such as spring break, the engine should support:
- entire break assigned to one caregiver
- split break into subranges
- odd/even year alternation
- fallback to base schedule if no break rule exists

## 8.4 Day-Type Priority

Recommended priority within school-specific logic:
1. full break interval rule
2. holiday-specific override
3. teacher planning / no-school day rule
4. early release modifier
5. base custody pattern

Note: holiday overrides may still sit above school rules globally depending on system precedence.

# 9. Approved Swap and Exception Logic

## 9.1 Swap as Structured Override

A swap request should not directly edit baseline rules. It should generate one or more ExceptionRule records after approval.

This preserves:
- auditability
- reversibility
- clean distinction between recurring rules and one-off changes

## 9.2 Partial-Day Swap Support

The engine must support partial intervals.

Examples:
- Friday 3 PM to Friday 8 PM
- Saturday overnight only
- exchange shifted from 5 PM to 8 PM

## 9.3 Swap Application Rules

After approval:
1. identify impacted generated schedule segments
2. split affected segments if needed
3. insert override segment(s)
4. regenerate handoff events
5. mark resulting segments as derived from approved swap

## 9.4 Counteroffer Handling

Countered swaps should not affect the actual generated schedule until approval.

Only final approved request states create exception rules.

---

# 10. Handoff Generation Logic

## 10.1 Handoff Trigger

A handoff occurs when responsibility moves from one caregiver to another across contiguous schedule segments.

## 10.2 Handoff Time Source

Handoff time should come from, in order:
1. explicit override on the relevant holiday/school/swap rule
2. child or household-level handoff rule
3. default schedule template value

## 10.3 Handoff Location Source

Location should come from:
1. explicit exception or swap setting
2. rule-level default
3. household default if available

## 10.4 Handoff Visibility

Generated handoffs should feed:
- Today + Handoff dashboard
- notifications
- calendar rendering
- activity history

---

# 11. Conflict Resolution Logic

## 11.1 Conflict Types

The engine should detect and classify:
- overlapping applicable rules
- missing anchor dates for alternating logic
- contradictory holiday assignments
- school event dates without clear assignment strategy
- duplicate exception coverage
- impossible handoff timing overlaps

## 11.2 Conflict Resolution Strategy

When possible, resolve automatically using precedence and priority.

When not possible:
- create a conflict state
- do not silently guess
- surface issue for user/admin review
- preserve source references for explanation

## 11.3 Explainability Requirement

For any schedule segment, the system should be able to explain:
- which rule won
- what was overridden
- whether a holiday or school event changed ownership
- whether a swap affected the final result

Example explanation:
"This Monday belongs to Parent B because the imported school calendar marked it as a teacher planning day, and the confirmed school-day rule assigns teacher planning days to Parent B."

---

# 12. Regeneration and Versioning

## 12.1 Regeneration Triggers

Regenerate the relevant schedule window when:
- a recurring rule changes
- a holiday rule changes
- school calendar events are confirmed or edited
- an approved swap is added or removed
- a manual override is created or revoked

## 12.2 Versioning

The engine should preserve rule versions or at least audit logs so the system can answer:
- what rule set produced this schedule previously
- when a schedule changed
- why a specific day changed ownership

## 12.3 Scope of Regeneration

Avoid regenerating the entire history by default.

Regenerate:
- future schedule window
- active planning window
- specific affected interval when possible

# 13. UX and Review States

## 13.1 Rule Statuses

Recommended statuses:
- draft
- proposed
- confirmed
- active
- superseded
- archived
- conflict

## 13.2 AI-Assisted Rule Review

If AI extracts rules from an agreement or school calendar, the extracted items should remain in `proposed` status until confirmed.

Only `confirmed` or `active` rules should affect generated schedules.

## 13.3 User Review Requirements

Critical fields that should be reviewable before activation:
- caregiver assignment
- date range or recurrence pattern
- year parity behavior
- handoff timing
- split intervals
- special exceptions

# 15. MVP vs Later Scope

## MVP Rules Engine Scope

Support these in MVP:
- fixed weekly patterns
- alternating weekends with explicit anchor date
- common rotating schedules if feasible
- holiday overrides
- school calendar event import after confirmation
- teacher planning day and no-school day assignment rules
- approved swap overrides
- handoff generation
- schedule conflict detection
- explainable source-of-truth outputs

## Later Scope

Add later:
- highly custom legal clause modeling
- fairness/balance analysis
- payments-linked obligations
- geographic travel logic
- GPS handoff verification
- court-certified evidence workflows
- advanced predictive scheduling intelligence

# 16. Example Resolution Walkthroughs

## Example A: Teacher Planning Day

Inputs:
- Base rule: Monday belongs to Parent A
- School import: Monday is teacher planning day
- School rule: teacher planning days go to Parent B

Result:
- Monday schedule segment assigned to Parent B
- handoff generated accordingly
- explanation references school rule override

## Example B: Alternating Thanksgiving

Inputs:
- Base rule gives Thursday to Parent A
- Holiday rule says Thanksgiving alternates by odd/even year
- Current year is even, assigned to Parent B

Result:
- Thanksgiving interval assigned to Parent B
- base rule overridden only for holiday interval

## Example C: Approved Partial-Day Swap

Inputs:
- Base weekend with Parent B
- Approved swap moves Friday 3 PM to 8 PM to Parent A

Result:
- original schedule segment split
- partial override inserted
- handoff events recalculated
- audit log references approved swap request
